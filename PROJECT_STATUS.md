# Project Status - Ryba

> Основной файл для отслеживания состояния новой версии проекта. Здесь фиксируется не только "что планируется", но и что именно считается завершением каждого этапа, какие артефакты должны появиться на выходе и по каким признакам можно понять, что этап действительно закрыт.

## Last Updated

`2026-04-01`

## Назначение файла

Этот файл нужен для того, чтобы проект не потерял форму по мере роста.

Он фиксирует:

- текущее состояние проекта;
- последовательность этапов;
- зависимость одного этапа от другого;
- конкретный результат, который должен появиться на выходе каждого этапа;
- критерии готовности;
- границы этапов;
- то, что нельзя считать выполнением этапа;
- риски, если этап будет сделан некачественно или не в правильном порядке.

Идея простая: Ryba нельзя развивать хаотично. У проекта высокая продуктовая сложность, поэтому нужен документ, который удерживает фокус разработки.

## Как читать этот файл

У каждого этапа есть:

- `Статус` — текущее состояние;
- `Цель` — зачем этот этап нужен;
- `Зависимости` — что должно быть готово до него;
- `Что делаем` — реальная зона работ;
- `Что должно получиться на выходе` — конкретный ожидаемый результат;
- `Как понять, что этап завершён` — критерии выхода;
- `Что не входит в этап` — защита от расползания scope;
- `Риски` — где этап чаще всего ломается.

## Легенда статусов

| Статус | Значение |
| --- | --- |
| `done` | Этап завершён и его результат присутствует в проекте |
| `in_progress` | Этап находится в активной работе |
| `planned` | Этап запланирован, но работа по нему ещё не началась |
| `blocked` | Этап нельзя продолжать без внешнего решения или без завершения зависимостей |
| `deferred` | Этап сознательно отложен на более поздний момент |

Это означает, что репозиторий уже содержит не только core-доменную базу, но и первую рабочую визуальную линзу поверх неё: канва больше не является демкой и использует реальные `spaces`, `entities`, `relations` и отдельный persistence layer для layout.

### Что это ещё не означает

Это всё ещё не значит, что проект готов как продукт для пользователей.
Пока ещё нет:

- production-реализации `web` и `api`;
- production-grade data layer с эксплуатационной наблюдаемостью и hardening;
- прикладных пользовательских сценариев;
- групп как subspaces;
- permission model;
- production-grade collaboration layer;
- типизированного entity detail/schema layer уровня S-4;
- документов, таблиц, groups/subspaces и permission model как рабочих продуктовых слоёв.

То есть продукт как система ещё не построен.
Построены стартовый каркас, технические прототипы, рабочее S-2 доменное ядро и базовая S-3 канва.

## S-3. Базовая Канва

**Статус:** `done`

### Цель

Сделать канву полноценным рабочим представлением сущностей и их связей.

### Зависимости

- завершён S-2;
- есть сущности, связи и backend-источник истины.

### Что делаем

- строим canvas view на React Flow;
- создаём entity nodes;
- создаём relation edges;
- сохраняем layout отдельно от core-данных;
- добавляем drag/drop, zoom, pan, selection;
- добавляем открытие detail view из node;
- добавляем создание сущности из canvas.

### Что должно получиться на выходе

Пользователь должен получить визуальное рабочее пространство, в котором он может:

- открыть space в режиме канвы;
- увидеть сущности как узлы;
- увидеть связи как ребра;
- перемещать сущности по канве;
- создавать новые сущности;
- создавать связи;
- переходить к записи объекта.

### Обязательные артефакты на выходе

- canvas route/view в `web`;
- node renderer;
- edge renderer;
- persistence для layout;
- привязка canvas к API;
- базовая модель canvas state.

### Как понять, что этап завершён

Этап завершён, если канва:

- не является демкой;
- отображает реальные сущности и связи;
- хранит layout отдельно от business truth;
- позволяет пользователю работать с данными визуально;
- не ломает основную доменную модель.

### Что не входит в этап

- сложный auto-layout;
- whiteboard-функциональность;
- realtime-синхронизация канвы;
- глубокая графическая полировка;
- mindmap-режимы;
- сложные анимации.

### Что должно выйти как главный результат

Главный результат этапа: канва становится рабочей навигационной линзой продукта.

### Риски

- превращение канвы в источник истины;
- смешивание visual state и domain state;
- желание сделать "красивую демку" вместо рабочего представления.

### Фактическое закрытие этапа (2026-04-01)

- `apps/api` получил отдельный canvas-модуль с `GET /spaces/:spaceId/canvas` и `PUT /spaces/:spaceId/canvas`;
- layout хранится в PostgreSQL в таблице `space_canvas_states` отдельно от `entities` и `relations`;
- `GET /spaces/:spaceId/canvas` возвращает пригодный к использованию state даже без предварительно сохранённого layout;
- `PUT /spaces/:spaceId/canvas` валидирует node/edge layout против реальных сущностей и связей пространства;

## S-7. Groups как Subspaces

**Статус:** `planned`

### Цель

Реализовать одну из главных отличительных идей проекта: подпространства как самостоятельные локальные контексты.

### Зависимости

- завершены S-3, S-4, S-5, S-6;
- существует уже несколько представлений, которые можно локализовать внутри group.

### Что делаем

- вводим `group` как отдельную сущность или особый контейнер;
- определяем правила вложенности;
- делаем переход внутрь group;
- делаем возврат наружу;
- локализуем внутри group сущности, views, документы и канву;
- проектируем containment logic.

### Что должно получиться на выходе

Пользователь должен уметь:

- создавать group внутри space;
- входить внутрь group;
- видеть, что у group есть свой локальный контекст;
- работать внутри group не как внутри папки, а как внутри подпространства.

### Обязательные артефакты на выходе

- модель `group`;
- правила принадлежности данных group;
- UI-навигация между outer space и inner group;
- локальные views внутри group;
- локальный document context;
- локальный canvas context.

### Как понять, что этап завершён

Этап завершён, если:

- group ощущается как отдельное рабочее пространство;
- внутри него существует собственная логика контекста;
- переход внутрь и наружу не ломает ментальную модель пользователя;
- фича добавляет новый класс работы, а не просто новый контейнер.

### Что не входит в этап

- бесконечная вложенность;
- сложные многослойные ACL внутри подпространств;
- чрезмерная универсализация containment logic.

### Что должно выйти как главный результат

Главный результат этапа: появляется signature feature проекта, отличающая Ryba от обычных таблиц и заметочников.

### Риски

- превратить group в папку;
- не продумать границу данных между outer и inner context;
- сделать навигацию слишком тяжёлой или неочевидной.

## Следующий реальный шаг после текущего состояния

Ближайший правильный ход:

1. перейти к S-4 и добавить entity detail + schema layer поверх уже рабочей S-3 канвы;
2. превратить inspector в полноценную detail view записи, а не только в минимальный side panel;
3. ввести первые entity types и field types без ухода в переусложнённую метасхему;
4. сохранить границу между visual state канвы и domain state предметной модели;
5. оставить Docker-first workflow и расширить smoke/integration coverage для нового S-4 слоя.

## Главный контрольный вопрос проекта

На каждом этапе нужно задавать один и тот же вопрос:

**Этот шаг укрепляет ядро Ryba или просто добавляет ещё одну красивую сложность?**

Если ответ второй, значит этап выбран неправильно или выполняется не в тот момент.
