# Ryba

> Новый старт проекта с нуля. Репозиторий фиксирует переписывание продукта с чистого листа, а старая версия считается устаревшей и больше не является источником истины.

## Статус проекта

Ryba сейчас находится в стадии полного перезапуска.

Цель этого репозитория не в том, чтобы хранить "какие-то остатки старого кода", а в том, чтобы с нуля собрать продукт с нормальной архитектурой, понятным стеком, ясным порядком разработки и чёткой границей между MVP и будущими расширениями.

Если раньше в репозитории уже была старая версия проекта, то для новой версии это не важно: текущий репозиторий должен отражать именно новую архитектурную линию.

Текущее состояние проекта, детализация этапов и критерии завершения каждого шага вынесены в отдельный файл [PROJECT_STATUS.md](./PROJECT_STATUS.md).

## Коротко: что такое Ryba

Ryba задумывается как визуальный knowledge + data + CRM workspace.

Это не просто CRM.
Это не просто заметочник.
Это не просто канва.
Это не просто таблицы.

Идея в том, чтобы объединить в одном продукте несколько режимов работы с информацией:

- структурированные сущности и связи между ними;
- визуальную канву для навигации по предметной области;
- документы и narrative-слой вокруг данных;
- табличные представления и сохранённые рабочие views;
- подпространства внутри пространства;
- позже внешние данные, запросы и CRM-сценарии как прикладной слой.

Проект должен дать пользователю возможность не просто хранить объекты, а реально работать с контекстом: связывать сущности, смотреть на них в разных представлениях, документировать решения, группировать контекст и со временем строить поверх этого рабочие сценарии команды.

## Главная продуктовая идея

Основная идея Ryba:

- у продукта есть ядро данных;
- поверх ядра есть разные линзы представления;
- документы и таблицы не существуют отдельно от сущностей;
- канва не является игрушкой, а служит интерфейсом навигации;
- группы внутри пространства должны работать как подпространства, а не как простые папки;
- CRM в этом проекте не является стартовой формой продукта, а появляется как прикладной слой поверх ядра.

Другими словами, сначала строится универсальная рабочая основа:

- workspace;
- space;
- entity;
- relation;
- views;
- documents;
- groups/subspaces;
- permissions;
- activity;
- external data;

И только после этого появляются специализированные сценарии вроде CRM.

## Что проектом не является

Важно зафиксировать, чем Ryba не должен становиться на старте:

- не очередным "конструктором всего" без твёрдого ядра;
- не spreadsheet-монстром;
- не перегруженной платформой микросервисов;
- не тяжёлой enterprise-системой до того, как появится рабочий core;
- не AI-first продуктом, который пытается компенсировать слабую архитектуру промптами;
- не очередной CRM-клон без собственной сильной идеи.

## Ключевой принцип разработки

Разработка идёт не по принципу "добавим понемногу всё", а по принципу:

**ядро данных -> базовый интерфейс -> рабочие представления -> подпространства -> внешние данные -> прикладные слои**

Это нужно, чтобы не получить красивую, но разваливающуюся систему.

В практическом виде правильный порядок такой:

1. техническая разведка;
2. core entities + relations;
3. canvas;
4. schema + detail view;
5. documents;
6. tables + saved views;
7. groups as subspaces;
8. permissions + activity;
9. realtime docs;
10. external data + queries;
11. CRM layer;
12. polish.

## Архитектурные принципы

Для новой версии проекта фиксируются следующие принципы:

- TypeScript почти везде и по возможности end-to-end;
- один основной runtime;
- один основной backend;
- одна основная база данных;
- модульный монолит вместо раннего микросервисного дробления;
- REST first, а не GraphQL ради GraphQL;
- строгая реляционная core-модель с контролируемой гибкостью через `jsonb`;
- канва не хранит источник истины, она хранит layout и визуальный контекст;
- документы и narrative-слой строятся вокруг сущностей, а не параллельно им;
- permissions-модель важнее, чем навороченная auth-модель;
- сложность добавляется только тогда, когда она оправдана реальным слоем продукта.

## Зафиксированный технологический стек

Ниже стек, который выбран как основа проекта.

### Общий фундамент

| Слой | Выбор | Почему |
| --- | --- | --- |
| Основной язык | TypeScript | Один язык для фронта, бэка, DTO, shared types, схем и части бизнес-логики |
| Runtime | Node.js 22.x | Современный runtime под актуальную Vite/Nest/TS toolchain |
| Репозиторий | Monorepo | Удобно делить `web`, `api`, `collab`, `ui`, `types`, `schemas`, `config` |
| Пакетный менеджер | pnpm workspaces | Удобная организация monorepo и зависимостей |
| Оркестрация задач | Turborepo | Быстрый запуск задач и кэширование для JS/TS-монорепы |

### Frontend

| Слой | Выбор | Почему |
| --- | --- | --- |
| UI framework | React | Основной слой клиентского интерфейса |
| Сборка | Vite | Быстрый dev-сервер и лёгкая сборка для app-like продукта |
| Язык | TypeScript | Строгая типизация и единый язык с backend |
| Роутинг | React Router | Баланс между простотой старта и постепенным ростом data-routing |
| Server state | TanStack Query | Кэширование, refetching и работа с async server-state |
| Local UI state | React state/context, позже точечно Zustand | Не перегружать приложение глобальным store раньше времени |
| Canvas | React Flow / xyflow | База под node-based editor, relations и визуальную навигацию |
| Таблицы | TanStack Table | Гибкий headless table layer без навязанного UI |
| Виртуализация | TanStack Virtual | Для длинных таблиц, списков и результатов запросов |
| Документы | Tiptap | Headless rich text editor, пригодный для mentions, embeds и дальнейшего collaboration |
| Runtime validation | Zod | Shared schemas, формы, query params, нормализация payload |

### Realtime и collaboration

| Слой | Выбор | Почему |
| --- | --- | --- |
| CRDT | Yjs | Основа для совместного редактирования и offline-capable sync |
| Collaboration server | Hocuspocus | Лучшее практическое сочетание для Tiptap + Yjs |

### Backend

| Слой | Выбор | Почему |
| --- | --- | --- |
| Framework | NestJS | Подходит под модульный монолит и сложную доменную модель |
| HTTP adapter | Fastify через Nest | Более быстрый вариант для backend-слоя |
| API стиль | REST first | Меньше лишней сложности на первом этапе |
| Документация API | Swagger / OpenAPI | Нужна для обозримого и проверяемого API-контракта |
| WebSockets | Точечно, по необходимости | Не тащить realtime везде, где можно начать с HTTP |

### Data layer

| Слой | Выбор | Почему |
| --- | --- | --- |
| Основная БД | PostgreSQL | Сильная реляционная база с `jsonb`, FTS и надёжной моделью данных |
| ORM для core app DB | Drizzle ORM | Типобезопасность и близость к реальному SQL |
| SQL/runtime слой для query service | `node-postgres` (`pg`) | Уместен для prepared queries, read-only connectors и data-source слоя |
| Поиск на MVP | PostgreSQL Full-Text Search | Без преждевременного ухода в Elastic/OpenSearch |

### Testing и tooling

| Слой | Выбор | Почему |
| --- | --- | --- |
| Unit/integration | Vitest | Естественно сочетается с Vite-экосистемой |
| E2E | Playwright | Надёжный путь для сложного UI, канвы и editor-сценариев |

### Осознанно не брать на day one

На старте проект сознательно не включает:

- Next.js / SSR-first подход;
- GraphQL как основной API;
- MongoDB как основную БД;
- микросервисы;
- Kafka / RabbitMQ;
- Elastic / OpenSearch;
- socket.io без жёсткой необходимости;
- тяжёлый enterprise auth suite;
- full BI/dashboard framework;
- ранний Redis "на всякий случай";
- AI-слой как часть базового продукта.

## Почему именно такой стек

У проекта высокая продуктовая сложность сама по себе.
Поэтому стек должен снижать трение, а не увеличивать его.

Выбор сделан по следующим практическим причинам:

- TypeScript end-to-end уменьшает разрыв между web и api;
- Nest позволяет держать сложную доменную модель в модульном монолите;
- PostgreSQL даёт надёжное ядро данных и при этом не ломается, когда нужен `jsonb`;
- Drizzle даёт контроль над SQL и не прячет реальную структуру запросов;
- React + Vite позволяют быстро двигаться в сложном UI;
- React Flow хорошо ложится на entity canvas;
- Tiptap даёт редактор, который можно реально адаптировать под сущности и ссылки;
- Yjs + Hocuspocus закрывают collaboration-сценарии документов без избыточной самодельщины;
- TanStack набором закрывает server-state, таблицы и виртуализацию;
- monorepo помогает держать web/api/shared-контракты в одной кодовой базе.

## Планируемая структура репозитория

Новая версия проекта строится как monorepo:

```text
.
├─ apps/
│  ├─ web/        # React + Vite клиент
│  ├─ api/        # NestJS backend
│  └─ collab/     # Hocuspocus / Yjs collaboration server
├─ packages/
│  ├─ ui/         # общие UI-компоненты
│  ├─ types/      # shared TypeScript types
│  ├─ schemas/    # shared Zod schemas
│  └─ config/     # общие конфиги: eslint, tsconfig и прочее
├─ .env.example
├─ .editorconfig
├─ .gitattributes
├─ .gitignore
├─ .nvmrc
├─ package.json
├─ pnpm-workspace.yaml
├─ turbo.json
└─ tsconfig.base.json
```

Эта структура выбрана не ради "красоты monorepo", а ради того, чтобы:

- фронт и бэк не жили как два случайных разных проекта;
- shared types и schemas были общими по-настоящему;
- collab server можно было развивать отдельно, но в том же репозитории;
- проект масштабировался по слоям, а не по хаосу.

## Базовые доменные сущности

На уровне продуктовой модели проект должен опираться на следующие ключевые сущности:

| Сущность | Роль в системе |
| --- | --- |
| User | Пользователь системы |
| Workspace | Верхний контейнер командной работы |
| Workspace Member | Связь пользователя с workspace и его роль |
| Space | Основное рабочее пространство внутри workspace |
| Group | Подпространство внутри space |
| Entity | Базовый объект предметной области |
| Entity Type | Определение типа сущности |
| Field / Property | Поля сущности |
| Relation | Связь между сущностями |
| View | Представление данных |
| Canvas View | Визуальное graph/canvas-представление |
| Table View | Структурированное табличное представление |
| Document View | Narrative/document-представление |
| Saved View | Сохранённая конфигурация представления |
| Data Source | Внешний источник данных |
| Saved Query | Сохранённый запрос |
| Dataset / Query Result | Результат выполнения запроса |
| Comment | Комментарии и обсуждение |
| Activity Event | Журнал действий |
| Permission Rule | Правила доступа |

## Как хранить данные

Хранение данных планируется так:

- фиксированные системные таблицы для пользователей, workspace, spaces, membership, entities, relations, views, permissions, comments, activities;
- schema/type definitions отдельным слоем;
- строгие ключевые поля в реляционных таблицах;
- более гибкие свойства частично через `jsonb`, но без потери управляемости;
- индексы и search-поля проектируются заранее, а не "как-нибудь потом".

Смысл в том, чтобы не скатиться ни в жёсткую негибкую схему, ни в бесконтрольный document store внутри Postgres.

## Важные архитектурные уточнения

### Канва

Канва не является источником истины.

Канва хранит:

- позиции узлов;
- размеры;
- визуальные группы;
- текущее выделение и локальный UI state;
- локальный layout.

Канва не должна становиться местом, где живёт вся бизнес-логика проекта.

### Документы

Документы нужны не как отдельный "модуль заметок", а как narrative-слой вокруг данных.

Документ должен уметь:

- содержать rich text;
- ссылаться на сущности;
- отображать mentions;
- строить backlinks;
- в будущем встраивать query results и previews.

### Таблицы

Table View нужен как business table, а не как бесконечная имитация Excel.

Таблицы должны покрывать:

- фильтры;
- сортировку;
- скрытие колонок;
- saved views;
- relation pickers;
- виртуализацию данных.

### Group как subspace

Group является одной из ключевых сильных идей проекта.

Group должен быть:

- не папкой;
- не просто тегом;
- не декоративной группировкой;
- а реальным подпространством со своим локальным контекстом.

Внутри group должны со временем жить:

- локальные entities;
- локальные views;
- локальные documents;
- локальная канва;
- локальные сценарии навигации.

## Порядок разработки по этапам

Ниже зафиксирован рекомендуемый порядок разработки. Он важнее списка "красивых фич", потому что определяет, выживет ли архитектура.

### Этап 0. Техническая разведка

**Цель:** снять технические риски и проверить основные библиотеки на совместимость с моделью продукта.

**Что делается:**

- поднимается monorepo;
- подключаются React + Vite + Nest + Postgres;
- отдельно проверяются React Flow, Tiptap, Yjs/Hocuspocus, TanStack Table;
- создаются маленькие технические прототипы вместо попытки сразу собрать продукт.

**Что должно получиться:**

- канва с кастомными node;
- базовый rich text editor;
- таблица с виртуализацией;
- минимальный realtime-прототип для текста.

**Критерий выхода:**

- понятно, как хранить entities;
- понятно, как хранить relations;
- понятно, как хранить canvas layout;
- понятно, как editor будет ссылаться на entities;
- понятно, какие библиотеки точно остаются.

**Чего не делать:**

- auth;
- permissions;
- CRM;
- красивые экраны;
- "продуктовый polish" вместо технической проверки.

### Этап 1. Core domain и backend-скелет

**Цель:** собрать минимальное доменное ядро.

**Базовые backend-модули:**

- auth;
- users;
- workspace;
- spaces;
- entities;
- relations.

**Первые таблицы:**

- users;
- workspaces;
- workspace_members;
- spaces;
- entities;
- entity_types;
- entity_properties;
- relations.

**Что должен уметь продукт:**

- регистрация пользователя;
- создание workspace;
- создание space;
- создание entity;
- связывание entity с другой entity;
- просмотр списка сущностей внутри space.

**Критерий выхода:**

- есть источник истины в entity;
- есть базовая relation model;
- есть CRUD API;
- есть базовая навигация по workspace и space.

**Чего не делать:**

- не строить "универсальный конструктор всего";
- не пытаться закрыть все будущие use-case до появления ядра.

### Этап 2. Базовая канва

**Цель:** сделать канву рабочей частью системы.

**Что делается:**

- canvas view;
- nodes для entity;
- edges для relations;
- отдельное хранение layout;
- drag/drop, zoom, selection.

**Что должен уметь продукт:**

- открывать space как canvas;
- добавлять существующую entity на канву;
- создавать entity прямо с канвы;
- проводить связь;
- переходить из node в detail view.

**Критерий выхода:**

- канва становится нормальной навигацией по предметной области, а не демо-игрушкой.

**Чего не делать:**

- сложный auto-layout;
- mindmap-фичи;
- лишние анимации;
- дорогой UI-polish раньше времени.

### Этап 3. Entity detail и schema layer

**Цель:** перестать работать с безликими объектами.

**Что делается:**

- вводятся entity types;
- вводятся field types;
- появляется detail/record view.

**Первые field types:**

- text;
- rich_text;
- number;
- boolean;
- date;
- select;
- multi_select;
- relation;
- user;
- url;
- status.

**Что должен уметь продукт:**

- создавать типы вроде Company, Contact, Task, Note, Project;
- задавать набор полей для каждого типа;
- открывать и редактировать detail view сущности.

**Критерий выхода:**

- entity перестаёт быть карточкой только с title и становится нормальным объектом предметной модели.

**Чего не делать:**

- formulas;
- rollups;
- automations;
- слишком сложную вычисляемую схему.

### Этап 4. Документы и narrative layer

**Цель:** добавить неструктурированный смысл вокруг данных.

**Что делается:**

- document/page view на базе Tiptap;
- rich text entity или document entity;
- entity mentions;
- backlinks;
- simple embeds;
- previews.

**Что должен уметь продукт:**

- создавать документ;
- писать текст;
- ссылаться на Company / Contact / Task;
- открывать связанные объекты из текста;
- видеть обратные ссылки.

**Критерий выхода:**

- продукт начинает выглядеть как knowledge workspace, а не только как registry объектов.

**Чего не делать:**

- comments по диапазонам текста;
- version history;
- слишком глубокий collaboration layer на уровне всего продукта.

### Этап 5. Таблицы и сохранённые views

**Цель:** дать пользователю структурированную рабочую линзу на сущности.

**Что делается:**

- table view;
- фильтры;
- сортировка;
- скрытие колонок;
- saved views;
- list view;
- basic detail side panel.

**Что должен уметь продукт:**

- таблица всех компаний;
- таблица задач по проекту;
- таблица контактов;
- отдельные сохранённые представления с фильтрами.

**Критерий выхода:**

- пользователь уже может реально работать с данными, а не только смотреть на канву и документы.

**Чего не делать:**

- не превращать table layer в Excel-клон.

### Этап 6. Group как subspace

**Цель:** реализовать ключевую отличительную идею проекта.

**Что делается:**

- вводится group entity;
- group получает собственный контекст;
- внутри group появляются локальные entities, views, documents и канва;
- делается вход внутрь group и возврат наружу;
- вводится containment logic.

**Что должен уметь продукт:**

- иметь space уровня "Sales";
- иметь внутри него group уровня "Enterprise clients";
- работать внутри этого group как в локальном подпространстве.

**Критерий выхода:**

- group ощущается как реальное подпространство, а не как папка.

**Чего не делать:**

- не делать глубокую бесконечную вложенность сразу;
- сначала один уровень, но хорошо.

### Этап 7. Permission model и activity

**Цель:** сделать продукт пригодным для командной работы.

**Что делается:**

- workspace membership;
- роли;
- доступ к space, group, entity, view;
- activity feed;
- audit events.

**Что должен видеть пользователь:**

- кто создал объект;
- кто изменил объект;
- кто переместил node;
- кто добавил relation;
- кто выдал доступ.

**Критерий выхода:**

- продукт уже можно показывать не только себе, но и маленькой команде.

**Чего не делать:**

- не строить сразу enterprise ACL-систему;
- сначала owner / editor / viewer и простые наследуемые правила.

### Этап 8. Realtime collaboration

**Цель:** добавить совместную работу там, где она действительно нужна.

**Что делается:**

- realtime для documents;
- presence / awareness;
- optimistic updates там, где это уместно;
- позже canvas sync.

**Приоритет:**

- сначала realtime в документах;
- потом, если действительно нужно, realtime на канве.

**Критерий выхода:**

- два пользователя могут редактировать документ одновременно без боли.

**Чего не делать:**

- не строить full-realtime-everything сразу.

### Этап 9. Data layer: внешние источники и запросы

**Цель:** подключить внешние данные безопасно и контролируемо.

**Что делается:**

- data source entity;
- подключение к Postgres;
- read-only credentials;
- saved query entity;
- query execution service;
- limits, timeout, logging;
- live table / dataset view;
- published outputs.

**Что должен уметь продукт:**

- сохранять запрос;
- выполнять его;
- показывать результат в table view;
- встраивать выборку в документ;
- переиспользовать dataset.

**Критерий выхода:**

- продукт становится knowledge + data workspace.

**Чего не делать:**

- не давать raw SQL отовсюду кому угодно;
- не разрешать запись во внешние БД.

### Этап 10. CRM-прикладной слой

**Цель:** собрать реальный CRM-сценарий поверх ядра.

**Что делается:**

- типы Company, Contact, Lead, Deal, Task, Meeting, Note;
- board view;
- calendar view;
- activity timeline;
- relation presets;
- CRM templates.

**Что должен уметь продукт:**

- вести компании;
- вести контакты;
- вести сделки;
- вести follow-up задачи;
- вести встречи;
- вести заметки;
- собирать pipeline-сценарии.

**Критерий выхода:**

- продукт уже можно честно использовать как lightweight CRM для команды.

**Чего не делать:**

- не начинать с CRM раньше, чем готово ядро.

### Этап 11. Полировка и зрелость

Сюда входят:

- поиск;
- история изменений;
- comments;
- notifications;
- import/export;
- file attachments;
- caching;
- UX polish;
- onboarding;
- empty states;
- templates.

Это не ядро, а усиление уже работающего продукта.

## Самые опасные ошибки для такого проекта

Есть несколько типичных мест, где такие продукты ломаются:

### 1. Слишком рано делают слишком много view

На старте достаточно:

- canvas;
- detail;
- document;
- table.

### 2. Слишком рано делают слишком гибкую схему

Ранний "конструктор всего" убивает темп и размазывает продукт.

### 3. Слишком рано делают CRM-специфику

CRM здесь должен быть приложением поверх ядра, а не самим ядром.

### 4. Слишком рано делают AI

AI не заменяет устойчивую доменную модель и не чинит слабую архитектуру.

### 5. Слишком рано делают внешние данные

Сначала внутренняя модель должна стать устойчивой, и только потом стоит подключать внешние источники.

## Почему PostgreSQL является ядром истины

PostgreSQL выбран как основная база по очень практическим причинам:

- у проекта много связей;
- у проекта много сущностей;
- у проекта будут membership, permissions, comments, activities, views, saved queries;
- нужна строгая модель данных;
- при этом нужна контролируемая гибкость через `jsonb`;
- на MVP нужен встроенный full-text search;
- в будущем есть смысл подключать внешние PostgreSQL data source.

Именно поэтому проект не стартует с document database.

## Почему backend строится как modular monolith

Ryba имеет достаточно сложную доменную модель и достаточно амбициозный продуктовый слой.
Если разбить это на микросервисы слишком рано, сложность только вырастет.

Модульный монолит здесь нужен, чтобы:

- сохранить целостность доменной модели;
- упростить разработку;
- не утонуть в сетевой и инфраструктурной сложности;
- оставить возможность аккуратно выделять отдельные сервисы позже, если это действительно понадобится.

На первом этапе логичный набор модулей такой:

- auth;
- identity;
- users;
- workspace;
- spaces;
- groups;
- entities;
- relations;
- views;
- documents;
- permissions;
- activities;
- data-sources;
- queries.

## Почему REST важнее GraphQL на старте

GraphQL здесь выглядит красиво только на бумаге.
На практике он слишком рано добавляет:

- сложность в permissions;
- сложность в nested loading;
- сложность в query cost;
- сложность в границах владения данными.

Поэтому проект стартует с:

- обычного REST API;
- хороших DTO;
- shared schemas;
- OpenAPI / Swagger.

Если позже появится реальная причина для GraphQL, его можно будет обсудить отдельно, но не раньше стабилизации core.

## Почему permissions важнее навороченной authentication

Для этого продукта главная сложность не в логине пользователя, а в доступе к:

- workspace;
- space;
- group;
- entity;
- view;
- query;
- dataset.

Именно поэтому на старте:

- auth остаётся простой;
- permission model проектируется заранее;
- роли и наследование доступа считаются важнее любых auth-экзотик.

## Как залить это в GitHub

Поскольку старая версия репозитория больше не актуальна, новая версия может стать полной заменой старого содержимого.

Базовый сценарий публикации:

```bash
git add .
git commit -m "chore: bootstrap new Ryba repository"
git push -u origin main
```

Если на удалённом репозитории уже лежит старая история и её нужно полностью заменить новой версией, то после осознанной проверки можно использовать:

```bash
git push -u origin main --force
```

Использовать `--force` нужно только если действительно принято решение, что старая версия репозитория больше не нужна.

## Что важно помнить дальше

У Ryba амбициозная продуктовая цель.
Поэтому основной риск проекта не в том, что "не хватит библиотек", а в том, что архитектура потеряет фокус.

Чтобы этого не произошло, дальше важно держаться нескольких правил:

- не плодить представления раньше времени;
- не делать CRM ядром продукта;
- не пытаться решить AI-слоем то, что должно решаться хорошей моделью данных;
- не тащить инфраструктуру без реальной необходимости;
- не размывать ядро ради "универсальности";
- сначала строить работающий core, потом расширять продукт.

## Итоговая жёсткая рекомендация

Если собрать всё в одну формулу, то базовая стратегия проекта такая:

**TypeScript end-to-end, Nest modular monolith, PostgreSQL as core truth, React/Vite on the frontend, Tiptap + Yjs for documents, React Flow for canvas, TanStack for structured views.**

Это не случайный набор технологий.
Это осознанный стек под продукт, в котором главная сложность находится в модели и взаимодействии слоёв, а не в количестве модных инструментов.
